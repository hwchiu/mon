DFW Controller — UI & API

The central brain: watches Zone / GroundRule / ZoneRule CRs, runs the Policy Engine, creates immutable PolicyVersion artifacts, pushes signed map data via gRPC, and exposes a live web UI + REST API for observation and configuration.

Web UI (port 8082)
  • Status cards: live counts of Zones, connected Agents (from gRPC Register/ReportStatus), PolicyVersions.
  • Agents table: AgentID, Zone, current Version, Healthy, LastSeen. Updated by distribution server.
  • Zones: list + "Add Zone" form (creates Zone CR via API).
  • Ground Rules Matrix: 4x4 (or N) interactive table. Click a cell to toggle allow/deny — it POSTs a GroundRule CR. The matrix reflects current CRs. This is the immutable baseline per the design.
  • Zone Rules: list + form to add DENY→ALLOW overrides (src/dst/ports/direction/action=allow).
  • Recent Policy Versions: the output of the engine (version, createdAt, groundHash).
  • Auto-refreshes every 10s. Fully client-side + calls the controller's /api/*.

Access: kubectl -n dfw-system port-forward svc/dfw-controller 8082:8082 then open http://localhost:8082

REST API (same port)

Lightweight JSON APIs used by the dashboard. All writes create the corresponding CRs so the ZoneReconciler + engine immediately see the change and produce a new PolicyVersion (which the gRPC layer pushes).

  • GET /api/status — counts + timestamp
  • GET /api/agents — live agents from gRPC (ID, zone, version, healthy, lastSeen)
  • GET /api/zones + POST /api/zones/create (body: Zone CR json)
  • GET /api/groundrules + POST /api/groundrules/create (the matrix toggle uses this)
  • GET /api/zonerules + POST /api/zonerules/create
  • GET /api/policies — list of PolicyVersion CRs

Example POST body for a GroundRule (see dashboard JS for exact shape used by the matrix):

{
  "apiVersion": "dfw.example.com/v1alpha1",
  "kind": "GroundRule",
  "metadata": { "name": "gr-zone-001-zone-002" },
  "spec": { "fromZone": "zone-001", "toZone": "zone-002", "ports": "all", "protocol": "all", "action": "allow" }
}

How configuration changes flow

  1. User (or dashboard) creates/updates a GroundRule or ZoneRule CR (via API or kubectl).
  2. ZoneReconciler (or any change) lists all Zones + GroundRules + ZoneRules.
  3. pkg/engine.CompileGroundAndZoneRules runs the 3-stage pipeline (collect → merge with ground-wins + OR zone rules → serialize to the 4 MapData structs + version + hashes). The ComputeVerdict implements the exact src_allows && dst_allows rule from the design.
  4. Controller creates a new PolicyVersion CR (immutable artifact).
  5. Because the reconciler holds a reference to the distribution server, it calls distServer.PushUpdate(..., json-serialized MapData, sig).
  6. Connected agents (via gRPC StreamUpdates) receive the update, verify (future), call Agent.ApplyPolicy which unmarshals the MapData and (in full impl) updates the eBPF maps + flips active_idx.
Ports on the controller pod/service:
8080 — controller-runtime metrics + (future) /metrics
8081 — healthz / readyz
8082 — this UI + the REST APIs above
9443 — gRPC Distribution (Register, StreamUpdates, ReportStatus)
9444 — webhook server (if webhooks are enabled)
