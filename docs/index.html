Project Overview

What is DFW?

A scalable, eBPF-based distributed firewall providing centralized policy management with decentralized enforcement across large, heterogeneous infrastructure.

Scale

100+ Kubernetes clusters and VMs spanning multiple network security zones, supporting 5,000+ servers from a single control plane.

Zero-Trust

Strict default-deny across all zone boundaries. All traffic between zones is blocked unless explicitly permitted by both source egress and destination ingress policies.

Real-Time

Policy propagation and enforcement within <1 second. eBPF map data updates are applied instantly — no recompilation, no restarts.

Design Principles

Four Core Principles

Default-Deny

Unlike Kubernetes (which defaults to allow), DFW enforces strict default-deny. All traffic between zones is blocked unless explicitly permitted by both the source egress and destination ingress policies.

Unidirectional

Policies flow only from the central controller to end-device agents. End-devices never send policy computation data back. Monitoring and audit data may flow upstream as it does not affect policy computation.

Zone-Centric

The entire IPv4 space is partitioned into non-overlapping network security zones. Every IP belongs to exactly one zone. Each Kubernetes cluster belongs to exactly one zone, but zones can span multiple clusters.

User-Agnostic

DFW does not design or manage user policies (NetworkPolicies, iptables, nftables). It ensures compatibility without interference. A (src, dst) pair is denied if any policy layer denies it — AND logic across DFW + user policies.

Data Flow

6-Step Policy Lifecycle

  1. 1
    Define

    Admin defines ground rules and zone rules in the Policy Registry.

  2. 2
    Compile

    Policy Engine merges rules using OR logic and serializes to eBPF map data structures.

  3. 3
    Assign Version

    An immutable version ID is assigned to each map data snapshot.

  4. 4
    Distribute

    Map data is pushed to all end-device agents in <1 second via the Distribution Channel.

  5. 5
    Apply

    Agent atomically updates eBPF maps — no recompilation, instant enforcement.

  6. 6
    Report

    Agent reports version, health, audit logs, and metrics back upstream (read-only feedback).

Zone Model

Network Security Zones

IP Partitioning
  • The entire IPv4 space 0.0.0.0/0 is partitioned into zones
  • Every IP belongs to exactly one zone
  • Each K8s cluster belongs to exactly one zone
  • Zones can span multiple clusters
  • Each zone has one or more non-overlapping CIDR blocks
Zone Properties
  • Non-overlapping CIDR assignments — no ambiguity
  • Zone membership is determined by IP address alone
  • Cross-zone traffic is governed by the ground rules matrix
  • Zone rules can selectively open denied paths
  • Zone rules cannot close allowed paths (OR logic)

Policy Rules Model

Policy Rules & Enforcement

Two layers of rules — ground rules (immutable baseline) and zone rules (selective overrides) — combine to produce the final enforcement verdict for every connection.

5.1 Ground Rules

Baseline Truth Table

Ground rules define the default allow/deny between every pair of zones. They form the baseline policy matrix and cannot be overridden to DENY by zone rules. Different port ranges can have different ground rules.

From ↓ / To → DMZ Internal Production Mgmt
DMZ ALLOW DENY DENY DENY
Internal ALLOW ALLOW DENY DENY
Production DENY DENY ALLOW DENY
Management ALLOW ALLOW ALLOW ALLOW
5.2 Zone Rules

Selective Overrides

  • Zone rules sit on top of ground rules
  • Can override DENY → ALLOW when ground rule is DENY
  • Cannot override ALLOW → DENY (ground rule always wins)
  • Multiple zone rules for the same (src, dst) are ORed together
  • Each rule: src_zone, dst_zone, ports, protocol, direction
5.5 OR Logic Examples
ScenarioRule 1Rule 2Verdict
Both allow ALLOW port 443 ALLOW port 80 ALLOW 443 & 80
Mixed ALLOW port 443 DENY port 80 ALLOW 443, DENY 80
Both deny DENY all DENY port 22 DENY all
5.3 Enforcement Logic

Connection Verdict

For a connection from source IP A to destination IP B on port P, both conditions must be met:

// Lookup zones
src_zone = get_zone(source_IP)
dst_zone = get_zone(dest_IP)

// Source egress check
source_allows = (
  ground_rule[src][dst][port] == ALLOW
  || any(zone_rules[src][dst][port] == ALLOW)
)

// Destination ingress check
dest_allows = (
  ground_rule[dst][src][port] == ALLOW
  || any(zone_rules[dst][src][port] == ALLOW)
)

verdict = source_allows && dest_allows
5.4 Policy Direction

Egress & Ingress

Egress Policy (Source Zone)
  • Defined by the source zone owner
  • Specifies allowed dst_zones, dst_ports, protocols
  • Multiple egress rules are OR'd
Ingress Policy (Destination Zone)
  • Defined by the destination zone owner
  • Specifies allowed src_zones, accepted ports, protocols
  • Multiple ingress rules are OR'd
A connection is allowed only if both egress AND ingress permit it. One denial is sufficient to block the connection.

Policy Engine

The Brain of DFW

Takes ground rules and zone rules as input, compiles them into optimized per-zone eBPF map data, and manages the complete policy lifecycle including versioning and conflict detection.

6.2 Compilation Pipeline

3-Stage Pipeline

1
Collect

Read ground rules, zone rules, and zone registry from the Policy Registry.

2
Merge & Validate

For each zone pair, combine ground rules with zone rules using OR logic. Detect conflicts — zone rules can override DENY → ALLOW but not ALLOW → DENY.

3
Serialize & Export

Serialize to map data structures, assign an immutable version ID, and push to the Distribution Channel.

6.3 Critical Design Decision

Data-Driven (Not Bytecode-Driven)

The controller never pushes eBPF bytecode. This is the core of the <1s propagation guarantee. Agents compile eBPF C source to bytecode once at boot, then read policy from eBPF maps at runtime.

PhaseWhoWhatFrequency
Boot Compile Agent (LLVM) C source → eBPF bytecode (data-driven program that reads maps) Once per boot
Policy Push Controller Map data only: zone-CIDR map, policy rules, allow/deny entries On change (<1s)
Runtime Enforcement Agent (kernel) eBPF program reads maps on each packet, instant verdict Per packet

No Bytecode Distribution

Smaller payload, no 100+ kernel variants to manage

New Kernel = Just Recompile

Agent restarts, compiles from C source — no binary update needed

Safe Hot Updates

Policy changes touch map data only — never compiled code

<1s Propagation

Map updates applied instantly — zero recompilation needed

6.5 eBPF Map Data Structures

Four Core Maps

zone_cidr_map
Key: zone_id  |  Value: CIDR prefix list
Zone-to-IP resolution — maps zone IDs to their CIDR ranges for fast lookup
policy_rule_map
Key: rule_key  |  Value: rule entry
Merged ground + zone rules — the compiled policy for each zone pair
verdict_map
Key: (src_zone, dst_zone, port)  |  Value: ALLOW / DENY
Data-driven verdict table — direct lookup for fast per-packet decisions
block_stats_map
Key: rule_key  |  Value: counter
Audit block statistics — per-rule drop counters for observability
6.6 Version Management

Immutable Versions

  • Every compilation produces an immutable version
  • Agents track their current version; engine pushes only when a newer version exists
  • Supports rollback to any previous version
  • Metadata: creation time, author, description
6.7 Conflict Detection

Safety Guarantees

  • Ground rules define the baseline — always respected, cannot be overridden to DENY
  • Zone rules can override DENY → ALLOW only
  • Conflicting zone rules are resolved with OR logic
  • System detects and warns on overlapping or redundant rules
Policy Propagation

How Policies Reach Agents

Strictly unidirectional. Policies flow from controller to agents only. Agents return read-only observability data — never policy computation inputs.

7.2 Allowed Upstream (Read-Only)

What Agents Can Send Back

  • Agent health status (alive / dead)
  • Policy version currently running
  • Policy version distribution statistics
  • Audit log entries (blocked traffic)
  • Performance metrics (propagation latency)
7.3 Never Allowed Upstream

What Agents Cannot Send

  • K8s NetworkPolicy data to controller
  • Policy computation requests
  • Zone topology queries
  • Rule modification requests
  • Any data affecting controller policy decisions
7.4 Propagation Architecture

Fan-Out Distribution

Fan-Out Distribution

Central controller distributes to zone-level distributors, which push to all agents in that zone. Multi-level fan-out reduces central bottleneck.

Delta Updates

Agents receive only delta changes from their current version, not the full policy. Minimizes bandwidth and application time.

Atomic Apply

New policy is atomically loaded on each agent. Either the old or new policy is always enforced — no intermediate inconsistent state.

Version Tracking
  • Each policy version is immutable and uniquely identifiable
  • System tracks which version each agent is running
  • Can identify agents that are out of sync
  • Retry with exponential backoff for failed deliveries
Security
  • Policy is encrypted and signed before distribution
  • Agents verify signature before applying any policy update
  • Fail-safe: agents retain last-known-good policy on failure
  • DFW runs before user policies in the network stack (outer security layer)
End-Device Agent

Kernel-Level Enforcement

A lightweight agent that compiles eBPF programs once at boot, then enforces centrally-distributed map data at the kernel level on every inter-zone packet — with zero knowledge of CNIs, pods, or overlay networks.

Interface-Scoped

Enforcement is limited to admin-specified host interfaces (e.g. eth0, bond0). CNI-specific interfaces (cilium_xxx, cni0, …) are never touched.

Inter-Zone Only

Only traffic crossing zone CIDR boundaries is in scope. Zone-internal traffic (same CIDR), pod-to-pod within a cluster, and overlay-internal frames (VXLAN, …) are out of scope.

Self-Compiling

Ships with C source code and an LLVM toolchain for on-the-fly eBPF compilation at boot. No pre-built bytecode required — kernel updates never cause compatibility issues.

8.2 Deployment

How Agents Are Deployed

Kubernetes (DaemonSet)

Deployed as a privileged DaemonSet pod with the required capabilities (NET_ADMIN, BPF, SYS_ADMIN, SYS_RESOURCE). Runs on every node in the target zone.

VM / Bare-Metal (Podman)

Run with Podman, configured via environment variables, with a persistent volume for the compiled eBPF program and last-known-good policy state.

8.5 Policy Application Flow

5-Step Apply Sequence

  1. 1
    Receive

    Agent receives map data from the Distribution Channel via the Upstream Connector.

  2. 2
    Verify

    Validates the digital signature and checks whether the version is newer than the currently running one. Stale or tampered updates are discarded.

  3. 3
    Update Maps

    Atomically swaps eBPF map contents using the double-buffer technique (active_idx flip). No bytecode reload — only data changes.

  4. 4
    Enforce

    The data-driven eBPF program reads the updated maps on each incoming/outgoing packet and returns an instant ALLOW or DENY verdict at the kernel level.

  5. 5
    Report

    Sends the new version confirmation, health status, and audit log entries back to the controller (read-only upstream feedback).

8.6 Enforcement Scope

What the Agent Enforces

In Scope
  • Inter-zone traffic on admin-specified host interfaces
  • Host-level source / destination IPs
  • Traffic crossing zone CIDR boundaries
  • Audit logging of blocked / matched packets
Out of Scope
  • Zone-internal traffic (same CIDR)
  • Pod-to-pod traffic within a K8s cluster
  • User-defined network policies
  • Overlay network internal traffic (VXLAN, Geneve, …)
  • CNI-specific interfaces (cilium_xxx, cni0, …)
