Independent Review

Design Review

Senior network engineer + SDN controller architect perspective on the current DFW design.

The zone-centric dual-consent model and data-driven eBPF approach are strong. This review surfaces the networking, cloud, and operational details that are easy to miss in the first clean architecture pass.

Top Gaps & Risks
  1. Bidirectional / Return Path Semantics
    Stateless design means return packets (SYN-ACKs, data, FINs) are evaluated as new flows. The current matrix examples do not show the four entries typically required for a working TCP session.
  2. Zone CIDR vs Real IPs (Node / Pod / Overlay)
    "Host-level source/destination IPs" + "every IP belongs to one zone" is clear in intent, but how pod CIDRs are folded in (especially Azure CNI overlay) is under-specified.
  3. verdict_map Scalability & Port Handling
    Direct (src_zone, dst_zone, port) lookup works for small cases but will explode with ranges or "all ports". Need clear strategy for wildcards + fallback to ordered scan.
  4. tc-BPF Attach Ordering & Coexistence
    Must run before CNI and host firewall. No concrete priority numbers, no logging of existing filters, no documented interaction with Azure accelerated networking.
  5. Distribution at Real Scale
    "Zone-X Distributor" is drawn but not designed. A single central gRPC pod will not comfortably handle thousands of persistent agent streams.
Priority Recommendations
  • Add explicit "full TCP session" examples (forward + return) and a symmetric: true helper in ZoneRule / engine.
  • Document exactly what must be in a Zone's cidrs for different CNI modes (and make the Azure test env prove it).
  • Define the eBPF map key structs and compiler emission rules (verdict vs policy_rule scan) before writing the C program.
  • Hard-code safe tc priorities in the agent and log tc filter show on the protected interfaces at startup.
  • Use the Azure multi-VNet test environment as the primary vehicle to validate the dual-consent model and propagation SLA under real conditions.
Other Notable Areas
  • Azure platform realities (NSG interaction, UDR/Firewall source IP rewriting, kernel matrix on AKS nodes)
  • Ground vs Zone rule governance & RBAC delegation
  • Low-level eBPF details (LPM layout, ringbuf payload, pinning, fragments/ICMP)
  • Testing strategy (how do we actually prove "both sides" with observable ringbuf events from src + dst agents?)
  • Operational runbooks (breakglass, partial rollout asymmetry, key rotation)
